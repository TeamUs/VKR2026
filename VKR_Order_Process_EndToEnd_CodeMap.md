## 6) Оплата вне приложения (manual/offline этап)

## 10) Ключевая модель статусов (чтобы не путать в ВКР)

В проекте используются **две независимые оси статусов**:

1. **Бизнес-статус заказа** (`orders.status`): `pending`, `paid`, `completed`, `cancelled`, ...  
   Меняется через `POST /api/admin/update-order-status`.

2. **Логистический статус доставки** (`delivery_tracking.status`): `Создан`, `Доставка внутри Китая`, ..., `Доставлен`  
   Меняется через `POST /api/admin/orders/:orderId/update-status`.

Это разделение важно: в коде явно указано, что авто-перевод `orders.status` в `completed` при `delivery_status = 'Доставлен'` отключен (см. комментарий в `backend/server.js`, строки `6238-6240`).

## 11) Формулировка процесса для пояснительной записки (готовый вариант)

Пользователь в Telegram Mini App переходит в раздел «Сделать заказ», заполняет многошаговую форму (товары, контакты, данные доставки) и отправляет заказ. Frontend формирует JSON-запрос к API `POST /api/orders`, после чего backend создает запись заказа в БД, добавляет позиции заказа, генерирует внутренний номер отслеживания и отправляет уведомление администратору в Telegram. Далее администратор связывается с пользователем, и оплата выполняется вне приложения. После подтверждения оплаты администратор переводит заказ в статус `paid` через админ-панель; backend обновляет статус и отправляет пользователю уведомление о смене статуса. Затем администратор по мере логистики обновляет этапы доставки (от «Создан» до «Доставлен») в модуле доставки. Пользователь в разделе «Отследить заказ» видит свои активные заказы, их текущий этап и может выполнять точечный поиск по трек-номеру, при этом backend проверяет принадлежность заказа пользователю по Telegram initData.
